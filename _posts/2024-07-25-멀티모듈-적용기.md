진행하고 있는 프로젝트에 멀티 모듈(Multi Module) 구조를 적용했다. 

먼저 왜 멀티 모듈 구조를 적용했는가? 부터 정리해야겠다. 

현재 프로젝트의 개발 프로세스는 이렇다.

develop 서버에서 개발 -> stage 서버로 배포 -> stage 서버에서 안정화된 버전을 production 서버로 배포

이러한 프로세스에서 단일 모듈로 구성된 프로젝트는 배포의 효율성이 좋지 않다.

예를 들면, production으로 배포한 버전에서 기획팀 쪽에서 관리자페이지의 새로운 기능을 추가하는 경우가 있었다.

하지만 단일 모듈 프로젝트는 관리자페이지의 기능만 수정할 수가 없고 다른 서비스 코드에 영향이 갔다.

또한 개발 쪽에서도 관리자페이지의 코드와 애플리케이션 API에서 사용하는 코드가 혼합되는 문제도 있고 의존성이 강했다.

또 배포에도 무거운 프로젝트를 매번 빌드하고 실행하다보니 배포 속도도 느렸다.

이러한 문제가 있기에 유연한 아키텍처 도입이 필요하다고 느껴졌다.

최근에는 MSA(마이크로서비스 아키텍처)로 구성한 프로젝트가 많고 해당 문제를 해결할 수는 있지만 지금 진행하고 있는 규모의 프로젝트에는 맞지 않는다고 생각했다.

또, 언젠가 서비스 규모가 커질 때 멀티 모듈 구조가 MSA로의 전환이 쉽다고 한다.
(멀티 모듈은 MSA로든 모놀리틱으로든 쉽게 바꿀 수 있는 구조여야 한다.)

그러면 멀티 모듈을 적용해보자.

먼저 이러한 단일 모듈 프로젝트가 있다.


```java
src
├─main
│  ├─java
│  │  └─com
│  │      └─example
│  │          └─demo

```

나는 core, api, admin, infrastructure, common 모듈로 나누었다.

각 모듈을 설명을 하자면

core : 핵심 도메인 로직이 들어 있는 모듈
api : 클라이언트와 통신하는 API 모듈
admin : 관리자페이지 API 모듈
infrasturcture : 외부 기능 모듈 (redis, feign을 이용한 외부 API 호출 등)
common : 모든 모듈에 사용하는 공통 코드(util 클래스 등)

구조를 만들어보면 이렇다.

api와 admin은 core 모듈을 의존한다.
